    Function Package#

    정의#

    Functional interfaces provide target types for lambda expressions and method references. Each functional interface has a single abstract method, called the functional method for that functional interface, to which the lambda expression’s parameter and return types are matched or adapted. Functional interfaces can provide a target type in multiple contexts, such as assignment context, method invocation, or cast context 출처 - https://docs.oracle.com/javase/8/docs/api/java/util/function/package-summary.html

    Java 1.8 version 부터 추가된 기능으로 Java에서 추상 method를 손쉽게 지원하기 위하여 구현된 Class이다! 단순하게 보면 Function을 변수로 담아내는 interface Class 정도로 이해할 수 있을 것 같다!

    요약#

    Functional Classparamreturn
    SupplierXO
    ConsumerOX
    Function<T,R>OO
    PredicateOboolean

    크게 4개의 Class가 존재한다. Function의 목적에 따라 위의 Class를 활용하여 작성한다!

    추가적으로 접두어에 Bi 를 붙여 매개변수가 2개인 Class를 사용할 수도 있다! (단, Supplier는 반환값의 type이 2개가 불가능 함으로 Bi Class가 존재하지 않는다)

    사용 예#

    생각보다 가까운 곳에 예제가 존재한다!

    package java.util.stream;

void forEach(Consumer<? super T> action);

Stream<T> filter(Predicate<? super T> predicate);

<R> R collect(Supplier<R> supplier,
                  BiConsumer<R, ? super T> accumulator,
                  BiConsumer<R, R> combiner);

    바로 Stream Class 이다! 해당 Functional interface를 통해 사용하는 곳에서 람다식으로 용도에 따라 method를 주입할 수 있다!

    내 사용 예#

    소스를 직접 작성하긴 애매하지만..! 작업 중 기존의 ExcelUtil에 file을 읽는 method가 활용도가 떨어져 따로 새로운 버전의 method를 작업했다! 헌데 공통적인 성향을 띄는 method임으로.. 추가로 유효성을 검증하는 부분을 Consumer interface로 구멍을 뚫어두었다! 하여 사용자는 필요 시 해당 param을 추가하여 data를 읽은 후 데이터를 사전에 검증할 수 있게 구현하였다!!

    data를 모두 읽은 후 검증 하는 것 보다 읽은 row 1개 씩 검증한다면 사용자는 검증에 통과한 row만을 받거나, 다른 row를 읽을 필요 없이 작업에서 나올 수 있다!

    내 생각#

    해당 기술이 너무 반가웠다! Java Script에서 활용하던 Function 변수와 같은 방식으로 코드를 구현할 수 있다는 점이 매우 친숙했다!! 공통 성향을 띄는 method를 개발하는 사람 입장에서는 해당 interface가 너무 반가웠을 것 같다!! 확실히 이 기능이 더해지고 나서 Java의 소스 작성 방식이 더욱 간결하고 직관적으로 변했다고 느꼈다! 특히 stream 문법이 그렇다! Function Package와 함께 등장하여 더욱 간결한 Java 언어 체계를 만들었다고 생각한다!! 확실히 기능적인 부분이나 동작에 이점이 있냐고 물어보는 개발자 에게는 시덥지 않은 Class로 보일 수 있겠지만.. 내가 볼 때는 확실히 함수형 개발의 패러다임을 만들어가는 시작점 아니였나 라고 보인다!! Java 같이 점유율이 높은 언어가 이러한 기능을 지원했다는 것은.. 큰 의미가 아닐까 싶다!! 😀😀

    설계도 처럼 요구사항이 명확한 구멍을 뚫어두는 행위는.. 가독성 뿐만 아니라 그 설계도를 읽고 작업하는 모든 작업자들이 공통적인 영역을 갖게 된다!! 하면 결국 코드 관리에 있어서도 확실한 장점 아닐까..? 하지만 재밌는 사실은 이렇게 interface 구멍을 뚫어 두어도.. 사용자는 정말 마음대로 작업한다는 것을 보았다!! 이래서.. 디자인 패턴을 추가적으로 사용하나 싶다.. 😫😫
