    Blocking Queue#

    정의#

    A Queue that additionally supports operations that wait for the queue to become non-empty when retrieving an element, and wait for space to become available in the queue when storing an element.

    다중 쓰레드 환경에서 동시성 처리를 위해 만들어진 인터페이스로, 대기하는 프로세스가 추가된 Queue 형태의 Class다!

    특징#

    • Wait
    • Null Element 사용 불가
    • Max Size 존재
      • Size가 가득 차 있을 시 해당 Thread는 공간이 생길 때 까지 대기한다. (blocking method 일 시)
    • Producer - consumer Pattern
      • 단, remove를 통해 특정 element를 제거 가능
    • Thread-safe
      • 단, 일반 동작 이외 경우에는 실패 가능성 존재 (add All 등)

    public class BlockingQueueExample {

    public static void main(String[] args) throws Exception {

        BlockingQueue queue = new ArrayBlockingQueue(1024);

        Producer producer = new Producer(queue);
        Consumer consumer = new Consumer(queue);

        new Thread(producer).start();
        new Thread(consumer).start();

        Thread.sleep(4000);
    }
}

public class Producer implements Runnable{

    protected BlockingQueue queue = null;

    public Producer(BlockingQueue queue) {
        this.queue = queue;
    }

    public void run() {
        try {
            queue.put("1");
            Thread.sleep(1000);
            queue.put("2");
            Thread.sleep(1000);
            queue.put("3");
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}

public class Consumer implements Runnable{

    protected BlockingQueue queue = null;

    public Consumer(BlockingQueue queue) {
        this.queue = queue;
    }

    public void run() {
        try {
            System.out.println(queue.take());
            System.out.println(queue.take());
            System.out.println(queue.take());
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}

    사용 예제#

    사용 예는 단순하다! Spring의 PostConstruct 를 통해 BlockQueue와 n개의 Thread를 생성하고, Thread는 while true 문 안에서 queue 값을 take 한다. interaface를 통해 Logging 요청이 들어올 시 queue에 put으로 적재하면 대기하던 thread가 값을 받은 후 logging 처리를 하게 구현되었다!

    내 생각#

    사실 처음에 내가 생각한 방식은 Spring의 Async 기능을 사용하려 했다! ThreadPool을 사용이 익숙했던 점도 있고, 그 방식이 조금 더 효율적일 것이라는 생각 때문이였다. 하지만 다른 분의 작업을 통해 Java의 고유한 다른 방식이 있다는 것을 알게 되었다!! 너무 재밌고 신기했다.. 사실 잘 몰라서 그렇지 Java에서도 요구사항에 따른 다양한 기능을 지원하고 있다는게 또 와닿았다!! 물론 Spring 처럼 쉽게 Annotation으로 구현되진 않지만, 잘 작성해두면 쉽게 효율적으로 의도한 방향에 맞추어 잘 사용할 수 있어 보였다! 한가지 궁금한 점은 ThreadPool 방식과 Blocking Queue 방식의 성능 차이점이였는데 알아본 결과 큰 성능의 차이는 없어 보였다! 단지 해당 task가 처리하는 작업의 양과 프로젝트의 하드웨어 성능 등이 변수가 되겠지만 두 방법 다 최적화를 한다면 성능은 동일할 것이라고 보는 것 같다! 단지 ThreadPool은 thread 수가 가변적이겠지만 BlockingQueue 방식은 초기에 지정한 thread로 동작한다는 차이점 정도가 있는 것 같다! (이 부분도 구현 방식에 따라 달라질 것 같긴하다)

    물론 Spring을 통해 쉽게 기능을 구현하는 것도 좋지만, 때로는 필요한 간단한 요구사항에 맞춰 위와 같은 기능을 이용하여 구현하는 역량도 필요하단 생각이 들었다. Java에 대해 잘 안다고 생각하는 일은.. 역시 앞으로도 없을 것 같다 😂😂
