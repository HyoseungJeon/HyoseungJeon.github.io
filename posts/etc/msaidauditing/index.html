    이전에 Scope 관련 글을 쓰며 추가적으로 내용을 찾아 AA 직무 다른 회사분에게 질문을 드렸는데, 해결 사항을 찾았다! 하여 간단하게 기록 해놓으려 한다~~

    개선 사항#

    이전에 Id 값을 매번 DTO에 set하여 보낸다고 하였다! 그래서 차선은 Context 정보가 있을 시 Header에 값을 세팅하여 요청을 보내는 방식을 생각하였다!

    해서 개인 마이그레이션 작업으로 하던 중 질문이 생겨 AA 분에게 질문을 드렸다! 헌데 Context가 아닌 Header 값을 통해 client Info를 관리하는 소스가 있다고 말씀해주시더라! 하여 내용을 찾아 보았는데 Id 값은 아니지만 locale, timeZone 등의 값을 Header X-ClientInfo 에 넣어 관리, 사용하고 있었다! 하지만.. Base DTO의 기본 생성자에 default로 값을 넣어주는 소스 때문인지 사용처가 없었다.. (변명이지만 그래서 못찾았다 😅)

    하여 결국 내 개발 방향과 딱 맞는 영역이 있었고, 해당 ClientInfo Entity에 userId Field도 추가하였다! 후에 Handling 하는 작업에 Context 값을 활용하여 넣는 소스를 추가 하였다!

    소스#

    <bind name="userInfo" value='@~(package)~.ClientInfoHolder@getClientInfo()'/>
~~~~
INSERT
#{userInfo.userId}

    결국.. 매우 간단하게 사용자 ID값을 mybatis에 Bind를 통해 간단하게 넣을 수 있게 되었다!! JPA의 Auditing에 나름 가까운 방향으로 잘 개발된 것 같다!! 이 bind 부분도 사실 지운 후에 DTO에 마지막에 set 하는 방식을 쓴다면 개발자는 Auditing과 같이 신경 쓸 부분이 아예 사라지겠지만..! 쓸데없이 중복된 데이터를 담아서 process가 흘러갈 필요가 없다고 생각되어 bind 방식으로 구현해두었다!! 위 영역도 mybatis 영역에 전역으로 설정할 수 있는 방식이 있는지.. 후에 더 찾아봐야겠다!!
